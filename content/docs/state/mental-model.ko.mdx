# 멘탈 모델

`@ilokesto/state`는 “또 다른 React 상태 관리 라이브러리”라고 생각하면 오히려 이해가 늦어집니다. 이 패키지는 React 안에 새로운 상태 모델을 만드는 쪽보다, 외부 스토어를 React에 자연스럽게 연결하는 쪽에 더 가깝습니다.

이 패키지는 store, hook, selector, writer라는 네 층으로 생각하면 가장 이해하기 쉽습니다.

## Provider + Context가 아니라 Store + Hook

Provider 중심 라이브러리와 달리, `@ilokesto/state`는 먼저 store를 두고 그 위에 React 훅을 얹습니다.

1. **store**는 React 바깥에 있고, 실제 상태를 소유합니다.
2. **hook**은 `create()`가 반환하며, 컴포넌트가 그 store를 구독하는 방식입니다.
3. **selector**는 컴포넌트가 읽을 범위를 좁힙니다.
4. **writer**는 여전히 전체 상태를 갱신하거나 reducer action을 dispatch합니다.

## 셀렉터가 핵심입니다

셀렉터는 raw external store보다 이 패키지를 더 실용적으로 만드는 핵심 요소입니다.

- 컴포넌트가 읽는 범위를 좁혀 주고,
- 불필요한 리렌더링을 줄이며,
- store snapshot이 깊게 같을 때 이전 선택 값을 재사용할 수 있게 합니다.

다만 selector가 있다고 해서 slice 전용 setter가 생기는 것은 아닙니다. writer는 여전히 전체 store state를 기준으로 동작합니다.

## 데이터 흐름

- **읽기**는 `useSyncExternalStore`를 통해 일어납니다.
- **쓰기**는 store writer 또는 reducer dispatch를 통해 일어납니다.
- **미들웨어**는 React 계층이 아니라 store 계층에 붙습니다.
- **SSR**은 `store.getInitialState()`를 server snapshot으로 사용하기 때문에 성립합니다.

## 왜 이 모양이 유용한가

이 구조는 중간 지점을 잘 잡아 줍니다.

- provider-heavy 구조보다 가볍고,
- 컴포넌트별 local state보다 명시적이며,
- 실제 store 위에서 동작하므로 미들웨어나 non-React 통합도 가능하게 남겨 둡니다.

정확한 API 모양을 보고 싶다면 [`create`](./create)를, 실제 조직 패턴을 보고 싶다면 [패턴](./patterns)을 이어서 읽어 보세요.
