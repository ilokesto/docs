# Middleware

`@ilokesto/state` ships middleware through the `@ilokesto/state/middleware` entrypoint. Middleware operate on the underlying `Store<T>` and return a new store with modified behavior.

Recent versions lean on the store's built-in middleware chain rather than treating middleware as one-off wrappers. In practice, that means middleware compose more naturally with the current `@ilokesto/store` runtime.

The usual pattern is:

```ts
import { create } from '@ilokesto/state';
import { logger, persist } from '@ilokesto/state/middleware';
import { pipe } from '@ilokesto/state/utils';

const store = pipe(
  { count: 0 },
  logger({ timestamp: true }),
  persist({ local: 'counter' }),
);

const useCounter = create(store);
```

## Call style

The built-in middleware generally support both:

- a **direct form**: `logger(store, options)`
- a **curried form**: `logger(options)(store)`

That makes them easy to combine with `pipe()`.

## Built-in middleware

### `logger`

Wraps `setState` and logs previous state, next state, and an action name. It also supports optional collapsed groups, timestamps, and diff output.

Current options are:

- `collapsed`
- `diff`
- `timestamp`

In production mode, it forwards updates without printing logs.

### `persist`

Loads initial state from configured storage and writes back after updates. The implementation supports storage options such as `local`, `session`, and `cookie`, depending on the config shape you pass.

Important details from the current implementation:

- it reads persisted state during store setup
- it writes back only when the persisted reference changes
- persisted payloads are stored as `{ state, version }`
- migrations are part of the persist config model
- migration execution currently applies only to `local` and `cookie` storage paths
- duplicate writes are skipped through a storage write cache

### `devtools`

Connects the store to the Redux DevTools browser extension.

Current behavior to know:

- browser-only (`window`-based)
- disabled in production
- handles `RESET`, `COMMIT`, and `ROLLBACK` dispatch messages
- sends updates as `${name}:${actionName}`

### `debounce`

Queues rapid updates and flushes them after the configured wait time. Function updaters are replayed in order against the current accumulated state.

### `validate`

Checks the next state against a Standard Schema v1 validator before committing the update.

Important limitation: async schema validation is rejected by the current implementation.

## Best practices

- Put middleware composition in one place so the store setup stays readable.
- Treat middleware order as part of the behavior of the store.
- Use `pipe()` when you want a left-to-right setup that is easy to scan.
- Prefer documenting your persistence storage choice explicitly so future migrations stay understandable.

## Caveats

- `devtools()` depends on the extension being available in the browser.
- `validate()` does not support async Standard Schema validation results.
- `persist()` behavior depends on the storage config you provide.
- `persist()` migrations do not currently run for `session` storage.
- `debounce()` delays writes by design, so do not use it where immediate synchronous state is required.
