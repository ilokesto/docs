# 구현 노트

이 페이지는 runtime이 실제로 어떻게 동작하는지 이해해야 할 때 중요한 현재 구현 세부 사항을 설명합니다.

## 하나의 runtime이 가진 요소들

각 `Toaster`는 다음을 포함하는 `ToastRuntimeApi` 하나를 만듭니다.

- toast store
- overlay store
- dismiss timer
- remove timer
- listener set
- runtime-local view config

## registry lookup

facade는 runtime state를 직접 들고 있지 않습니다. `toasterId`로 registry에서 runtime을 찾습니다.

해당 runtime이 마운트되어 있지 않으면 `requireRuntime()`이 오류를 던집니다.

## overlay presence mirror

toast가 추가되면 runtime은 같은 id를 사용해 overlay store에도 presence를 보장합니다.

- toast 추가 -> overlay presence 보장
- toast dismiss -> overlay presence close
- toast remove -> overlay presence remove

overlay 계층은 presence lifecycle 용도이고, toast 정책 자체를 담당하지는 않습니다.

## timer와 pause 로직

- visible toast는 duration에서 누적 pauseDuration을 뺀 값으로 dismiss timer를 예약하고
- dismiss는 delayed removal을 예약하며
- hover pause는 dismiss timer를 지우고 pause 시작 시각을 기록하고
- resume은 누적 pauseDuration을 기준으로 dismiss timing을 다시 계산합니다

## visible filtering

`limit`과 `position`은 runtime view filter입니다. raw store entry를 삭제하지는 않습니다.

## promise 전환

`toast.promise()`는 먼저 loading toast를 만들고, 그 다음 같은 id를 재사용해 success 또는 error로 바꿉니다.
