# `createProxy`

`createProxy`는 `Show.div`, `For.ul`, `Mount.section` 같은 proxy-backed export를 만드는 함수입니다. 직접 비슷한 component family를 만들고 싶다면 이 페이지가 핵심입니다.

`createProxy`가 처음에 어렵게 느껴지는 이유는 **runtime 동작**과 **type-level 동작**이 한곳에 섞여 있기 때문입니다.

- runtime에서는 `.div`, `.Link` 같은 속성을 해석하는 `Proxy`이고,
- type level에서는 `ProxyType`, `UtilinentRegister`를 통해 그 속성들이 타입으로 보이게 만듭니다.

그래서 이해 순서도 runtime부터, 타입은 그다음이 좋습니다.

## `createProxy`가 하는 일

실제 시그니처는 다음과 같습니다.

```ts
createProxy(base, renderForTag, category)
```

이 함수는 세 가지를 하나의 export로 합칩니다.

1. **base component**
2. `.div`, `.button`, `.section` 같은 **intrinsic tag variant**
3. `PluginManager`를 통해 해석되는 **plugin-backed variant**

그래서 결과적으로 하나의 export가 “컴포넌트”이면서 동시에 “속성을 가진 객체”처럼 동작합니다.

## 세 개의 입력값

### `base`

사용자가 proxy 자체를 직접 호출했을 때의 기본 컴포넌트입니다.

예를 들어 `Show`의 base form은:

```tsx
<Show when={user}>...</Show>
```

### `renderForTag`

사용자가 `.div`, `.button` 같은 속성을 요청했을 때 어떻게 렌더링할지를 결정하는 factory입니다.

보통 `forwardRef` wrapper를 반환해서, 요청된 태그에 커스텀 동작을 입힌 형태로 렌더링합니다.

### `category`

plugin lookup이 먼저 보는 bucket입니다. 사용자가 알 수 없는 속성에 접근했을 때, `PluginManager`에서 어떤 카테고리를 먼저 찾을지를 결정합니다.

내장 카테고리에는 `show`, `for`, `repeat`, `mount`, `switch`, `base`가 있지만, runtime에서는 custom string category도 사용할 수 있습니다.

## 해석 순서

`Clickable.div`나 `Clickable.Link` 같은 속성에 접근하면, 프록시는 다음 순서로 값을 찾습니다.

1. target에 이미 존재하는 속성
2. 현재 `category`에 등록된 plugin
3. `base`에 등록된 plugin
4. `undefined`

첫 번째 규칙이 생각보다 중요합니다.

- HTML tag variant는 이미 existing property로 들어가 있고,
- plugin variant도 한 번 해석되면 target에 캐시되어 이후에는 existing property가 됩니다.

즉 실질적으로는 **built-in이 먼저, 그다음 category plugin, 그다음 base plugin, 그리고 성공한 plugin lookup은 캐시된다**고 이해하면 됩니다.

## 가장 작은 runtime 모델

타입 얘기를 하기 전에, 먼저 runtime에서 어떤 모양을 만들고 있는지부터 보면 이해가 쉽습니다.

```tsx
import { ComponentPropsWithRef, createElement, forwardRef } from 'react';
import { createProxy } from '@ilokesto/utilinent';

type ClickableProps = {
  active?: boolean;
  children?: React.ReactNode;
};

function BaseClickable({ children }: ClickableProps) {
  return <>{children}</>;
}

const renderForTag =
  (tag: any) =>
  forwardRef(function Render(
    { active = false, children, ...props }: ClickableProps & ComponentPropsWithRef<any>,
    ref: any,
  ) {
    return createElement(tag, {
      ...props,
      ref,
      'data-active': active,
    }, children);
  });

const Clickable = createProxy(BaseClickable, renderForTag, 'clickable');
```

여기서 runtime 관점의 핵심은 이미 완성됩니다.

- `Clickable` 자체가 base form이고,
- `Clickable.div`, `Clickable.button`은 HTML tag variant이며,
- `Clickable.SomePlugin`은 나중에 `PluginManager`를 통해 붙일 수 있습니다.

## 끝까지 따라가는 예제: `Clickable` 만들기

가장 짧은 완성형 튜토리얼은 이 흐름입니다. 목표는 다음과 같은 component family를 만드는 것입니다.

- base form인 `Clickable`
- intrinsic variant인 `Clickable.button`
- 나중에 plugin으로 붙는 `Clickable.Link`

### 1단계: props와 base behavior 정의하기

먼저 모든 variant가 공유할 기본 동작부터 잡습니다.

```tsx
type ClickableProps = {
  active?: boolean;
  children?: React.ReactNode;
};

function BaseClickable({ children }: ClickableProps) {
  return <>{children}</>;
}
```

이 예제에서는 base form 자체는 거의 아무 일도 하지 않습니다. 보통 중요한 부분은 각 tag variant가 어떻게 렌더링되는지에 있습니다.

### 2단계: tag variant가 렌더링되는 방식을 정의하기

`renderForTag`는 family-wide behavior를 각 intrinsic element에 붙이는 곳입니다.

```tsx
import { ComponentPropsWithRef, createElement, forwardRef } from 'react';

const renderForTag =
  (tag: any) =>
  forwardRef(function Render(
    { active = false, children, ...props }: ClickableProps & ComponentPropsWithRef<any>,
    ref: any,
  ) {
    return createElement(tag, {
      ...props,
      ref,
      'data-active': active,
    }, children);
  });
```

이 부분이 있어야 `Clickable.button`과 `Clickable.a`가 서로 unrelated component가 아니라 하나의 family처럼 느껴집니다.

### 3단계: proxy 만들기

```tsx
const Clickable = createProxy(BaseClickable, renderForTag, 'clickable');
```

runtime에서는 이 한 줄로 다음이 가능해집니다.

```tsx
<Clickable>Plain base form</Clickable>
<Clickable.button active>Save</Clickable.button>
<Clickable.a href="/docs">Read docs</Clickable.a>
```

### 4단계: 타입 계층 붙이기

runtime만 놓고 보면 여기서 이미 동작하지만, TypeScript가 전체 surface를 이해하려면 `BaseTypeHelperFn`, `ProxyType`가 필요합니다.

```tsx
import { BaseTypeHelperFn, ProxyType } from '@ilokesto/utilinent';

type BaseClickableType<X = object> = {
  (props: X & ClickableProps): React.ReactNode;
};

interface BaseClickableTypeFn extends BaseTypeHelperFn {
  type: BaseClickableType<this['props']>;
}

type ClickableType = ProxyType<BaseClickableTypeFn, 'clickable'>;

export const Clickable: ClickableType = createProxy(BaseClickable, renderForTag, 'clickable');
```

### 5단계: plugin-backed variant는 나중에 붙인다

family가 준비된 뒤에 plugin 등록은 별도의 문제로 다룹니다.

```tsx
import Link from 'next/link';
import { PluginManager } from '@ilokesto/utilinent';

PluginManager.register({
  clickable: {
    Link,
  },
});
```

이제 `Clickable.Link`를 이름으로 해석할 수 있습니다.

여기서 기억할 구분은 분명합니다.

- `renderForTag`는 intrinsic tag variant의 동작을 정의하고,
- `PluginManager`는 추가 named variant의 존재를 결정합니다.

### 6단계: TypeScript에도 plugin variant를 알려준다

```ts
declare module '@ilokesto/utilinent' {
  interface UtilinentRegister {
    clickable: {
      Link: typeof Link;
    };
  }
}
```

이 단계가 없으면 runtime은 동작해도 TypeScript는 계속 오류처럼 보일 수 있습니다.

## 타입을 붙이는 방법

위 예제는 runtime에서는 동작하지만, TypeScript가 원하는 shape를 알기 위해서는 타입 설명이 더 필요합니다.

패키지 내부 컴포넌트들도 같은 패턴을 따릅니다.

```tsx
import { ComponentPropsWithRef, createElement, forwardRef } from 'react';
import { BaseTypeHelperFn, ProxyType, createProxy } from '@ilokesto/utilinent';

type ClickableProps = {
  active?: boolean;
  children?: React.ReactNode;
};

function BaseClickable({ children }: ClickableProps) {
  return <>{children}</>;
}

const renderForTag =
  (tag: any) =>
  forwardRef(function Render(
    { active = false, children, ...props }: ClickableProps & ComponentPropsWithRef<any>,
    ref: any,
  ) {
    return createElement(tag, {
      ...props,
      ref,
      'data-active': active,
    }, children);
  });

type BaseClickableType<X = object> = {
  (props: X & ClickableProps): React.ReactNode;
};

interface BaseClickableTypeFn extends BaseTypeHelperFn {
  type: BaseClickableType<this['props']>;
}

type ClickableType = ProxyType<BaseClickableTypeFn, 'clickable'>;

export const Clickable: ClickableType = createProxy(BaseClickable, renderForTag, 'clickable');
```

중요한 것은 helper 이름을 외우는 게 아니라 의미를 이해하는 것입니다.

- `BaseTypeHelperFn`은 “어떤 props가 들어오면 어떤 callable component type이 나와야 하는가”를 표현하는 계약이고,
- `ProxyType<F, Category>`는 “그 callable type에 HTML tag와 plugin key를 추가한다”는 뜻이며,
- category string은 나중에 plugin을 어떤 bucket에서 찾을지와 연결됩니다.

## custom variant 등록하기

proxy가 준비되면 `PluginManager`로 새 속성을 붙일 수 있습니다.

이 단계는 intrinsic tag 생성과는 다릅니다. HTML tag variant는 `renderForTag`를 통해 미리 만들어지지만, plugin-backed variant는 나중에 `PluginManager`에서 이름으로 해석됩니다.

```tsx
import Link from 'next/link';
import { PluginManager } from '@ilokesto/utilinent';

PluginManager.register({
  clickable: {
    Link,
  },
});
```

그러면 runtime에서는 다음처럼 plugin-backed property lookup이 가능해집니다.

```tsx
<Clickable.Link href="/login">
  Login
</Clickable.Link>
```

다만 `active` 같은 family-specific prop이 plugin-backed variant에도 그대로 의미가 있다고 자동으로 가정하면 안 됩니다. 그 부분은 결국 `renderForTag`와 등록한 컴포넌트가 어떻게 상호작용하도록 설계되었는지에 달려 있습니다.

## 타입도 같이 등록하기

runtime 등록만으로는 충분하지 않습니다. TypeScript가 `Clickable.Link`를 이해하려면 `UtilinentRegister` 확장이 필요합니다.

```ts
import Link from 'next/link';

declare module '@ilokesto/utilinent' {
  interface UtilinentRegister {
    clickable: {
      Link: typeof Link;
    };
  }
}
```

이제 `ProxyType<..., 'clickable'>`가 등록된 키를 타입 수준에서도 볼 수 있습니다.

## `base` 카테고리는 언제 쓰나

`base` 아래에 등록하면, **모든 proxy family**가 자기 category에서 찾지 못한 이름을 `base`에서 다시 찾게 됩니다.

강력하지만 범위가 넓습니다. 정말 공통적인 variant일 때만 `base`를 쓰는 편이 좋습니다.

한 family에만 속하는 variant라면 custom category를 쓰는 편이 더 안전합니다.

## 흔한 실수

### 1. 타입부터 먼저 이해하려고 하는 것

`ProxyType<this['props']>` 같은 문법이 먼저 보이면 이해가 막히기 쉽습니다. 이럴 때는 runtime 흐름으로 돌아가면 됩니다.

`base -> renderForTag -> createProxy -> property access -> plugin lookup`

이게 실제 모델입니다.

### 2. built-in보다 plugin이 먼저라고 생각하는 것

이미 target에 있는 속성은 plugin lookup보다 먼저 반환됩니다.

### 3. runtime plugin은 등록했는데 타입은 안 붙이는 것

이 경우 런타임에서는 동작해도 TypeScript는 계속 깨진 것처럼 보일 수 있습니다.

### 4. `base`를 너무 쉽게 사용하는 것

`base`는 편리하지만 영향 범위가 너무 넓어질 수 있습니다.

## 언제 `createProxy`를 써야 하나

다음과 같은 경우에 잘 맞습니다.

- 하나의 base primitive에 여러 tag variant가 필요할 때
- category-specific plugin extension이 필요할 때
- `Thing.div`, `Thing.Link` 같은 API 모양이 중요한 경우

반대로 일반 컴포넌트 하나면 충분한데도 `createProxy`를 쓰는 것은 과합니다. 이 함수는 “래퍼 하나 더 만들기”보다 **component family 만들기**에 더 가깝습니다.
