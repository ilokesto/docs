# 멘탈 모델

`@ilokesto/toast`는 세 층으로 생각하면 가장 이해하기 쉽습니다.

- `toast` facade
- provider-scoped runtime
- overlay 기반 presence lifecycle

## `toasterId` 기준 runtime registry

암묵적인 전역 싱글턴은 없습니다. 각 `Toaster`가 `toasterId`별로 runtime 하나를 등록합니다.

`toast.success()`를 호출하면 facade는 먼저 registry에서 runtime을 찾습니다.

## facade / runtime 분리

import한 `toast` 객체는 상태를 직접 소유하지 않습니다. 명령만 라우팅합니다.

실제 runtime이 소유하는 것은 다음입니다.

- item state
- timer
- dismiss/remove 동작
- visible filtering
- pause/resume 동작

## store + overlay presence 분리

각 runtime은 두 가지를 함께 씁니다.

- toast 데이터와 timer를 위한 toast store
- presence lifecycle 전용 overlay store

그래서 내부적으로 overlay를 쓰더라도, toast는 더 높은 수준의 패키지처럼 느껴집니다.

## 옵션 병합 순서

실제 옵션은 다음 순서로 병합됩니다.

1. global `toastOptions`
2. per-type `toastOptions.success/error/loading/blank/custom`
3. per-toast call option

## dismiss와 remove의 차이

- `dismiss()`는 soft close + delayed removal이고
- `remove()`는 즉시 삭제입니다

애니메이션과 lifecycle 타이밍 때문에 이 차이가 중요합니다.

정확한 공개 표면을 보려면 [API 레퍼런스](./api), 렌더링 세부 사항을 보려면 [Toaster](./toaster)로 이어서 읽으면 됩니다.
