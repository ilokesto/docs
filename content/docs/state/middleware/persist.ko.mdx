# 영속화

`persist` 미들웨어는 브라우저 세션 전반에 걸쳐 스토어 상태를 자동으로 저장하고 복원합니다. 이는 인메모리 스토어와 영구 저장소 대상을 연결하는 다리 역할을 합니다.

## 사용법

`persist`는 스토어 객체에 직접 적용하거나, 스토어 생성 시 커링(currying)된 미들웨어 팩토리로 사용할 수 있습니다.

```ts
import { persist } from '@ilokesto/state/middleware';

const store = persist({ local: 'counter-storage' })({ count: 0 });
```

## 저장소 구성

미들웨어는 설정 객체를 통해 세 가지 주요 저장소 대상을 지원합니다.

- **`local`**: (string) `localStorage`에 저장할 키 이름입니다. 브라우저가 꺼졌다가 켜져도 유지되어야 하는 데이터에 사용하십시오.
- **`session`**: (string) `sessionStorage`에 저장할 키 이름입니다. 탭이 닫힐 때 삭제되어야 하는 데이터에 사용하십시오.
- **`cookie`**: (string) 쿠키 저장소에 저장할 키 이름입니다. 서버와 상태의 일부를 공유해야 할 때 유용합니다.

## 복원 흐름 (Rehydration Flow)

`persist`와 함께 초기화된 스토어는 다음과 같은 설정 순서를 따릅니다.
1. **초기 설정**: 제공된 초기 상태로 스토어가 생성됩니다.
2. **동기 읽기**: 미들웨어는 즉시 설정된 저장소로부터 영속화된 상태를 읽어오려고 시도합니다.
3. **교체**: 유효한 데이터가 발견되면 그 복원된 값이 스토어의 현재 상태가 됩니다.
4. **참조 변경 가드**: 영속화된 상태가 초기 상태와 다를 때만 스토어의 참조가 업데이트됩니다.

## 저장소 형태 및 버전 관리

데이터는 다음과 같은 래퍼 객체 안에 저장됩니다.
```json
{
  "state": { ... },
  "version": 2
}
```

- **버전 계산**: 현재 모델에서 `version`은 `migrate` 체인의 길이에 따라 결정됩니다. 마이그레이션 함수가 3개라면 버전은 3이 됩니다.
- **Write-Through 동작**: 스토어 커밋이 발생할 때마다 미들웨어는 새로운 상태를 저장소에 씁니다.
- **쓰기 캐시 (Write Cache)**: 불필요한 I/O를 피하기 위해 미들웨어는 인코딩된 persisted payload를 내부적으로 캐시하고, 다음 인코딩 결과가 같으면 저장소 쓰기를 생략합니다.

## 마이그레이션 (Migrations)

`migrate` 옵션은 오래된 상태 형식을 현재 형식으로 변환하는 함수들의 배열을 받습니다.

- **대상 제약**: 현재 마이그레이션은 `local` 및 `cookie` 대상에서만 지원됩니다. `session` 저장소는 마이그레이션 흐름을 지원하지 않습니다.
- **점진적 업데이트**: 배열의 각 함수는 하나의 버전 단계를 나타냅니다. 저장된 버전이 1이고 현재 체인 길이가 3이라면, 함수 `[1]`과 `[2]`가 차례대로 적용됩니다.

## 쿠키 제약 사항

`cookie` 대상을 사용할 때는 다음을 주의하십시오.
- **크기**: 쿠키는 일반적으로 4KB로 제한됩니다. 거대한 상태 트리는 영속화에 실패할 수 있습니다.
- **직렬화**: 데이터는 JSON으로 직렬화됩니다. 상태에 순환 참조나 직렬화할 수 없는 객체(예: 함수 또는 클래스)가 포함되지 않도록 하십시오.

## 미들웨어 순서

미들웨어 체인 내에서 `persist()`의 위치는 중요합니다.

**권장 순서**: `persist()`를 체인의 **끝부분**에 배치하십시오.
이렇게 해야 `validate`나 `debounce` 같은 다른 미들웨어가 업데이트를 처리한 후의 "최종" 상태만 저장소에 저장됩니다.

```ts
const store = pipe(
  validate(schema),
  persist({ local: 'app-state' })
)({ count: 0 });
```
