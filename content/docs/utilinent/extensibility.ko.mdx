# 확장성

이 페이지는 일반적인 소비자가 직접 만지지 않을 수도 있는 Utilinent의 내부 메커니즘을 설명합니다. proxy-backed variant와 plugin 기반 확장을 가능하게 하는 부분입니다.

## `createProxy`

`createProxy(base, renderForTag, category)`는 `Show`, `For` 같은 proxy-backed export를 만드는 함수입니다.

이 함수는 다음 요소를 바탕으로 target을 구성합니다.

- base component
- 생성된 HTML-tag variant
- 접근 시점에 해석되는 plugin-backed variant

해석 순서는 중요합니다.

1. target에 이미 존재하는 속성
2. 현재 category에 등록된 plugin
3. `base`에 등록된 plugin

즉 built-in이 항상 plugin lookup보다 먼저 우선합니다.

## `PluginManager`

`PluginManager`는 plugin-backed variant 뒤에 있는 singleton registry입니다.

```ts
import { PluginManager } from '@ilokesto/utilinent';

PluginManager.register({
  show: {
    FancyCard,
  },
  base: {
    MotionButton,
  },
});
```

공개 static 메서드는 다음과 같습니다.

- `register()`
- `get()`
- `getAll()`
- `has()`
- `unregister()`

이 객체는 singleton global state이기 때문에, 한 번 등록된 plugin은 이후의 proxy lookup 전반에 영향을 줍니다.

## export된 타입

루트 패키지는 확장 작업을 위한 타입 헬퍼도 함께 export합니다.

- `UtilinentRegister`
- `BaseTypeHelperFn`
- `ProxyType`

이 타입들은 typed module augmentation이나 proxy-aware API를 만들고 싶을 때 의미가 있습니다.

## `useIntersectionObserver`는 어디에 들어가나

이 hook은 사용자도 직접 쓸 수 있지만, `Observer`와 `Slacker`가 이 훅 위에서 동작하기 때문에 확장성 맥락에서도 중요한 위치를 차지합니다.

## 언제 이 페이지를 읽으면 좋은가

Utilinent 자체를 확장하고 싶을 때 이 페이지를 읽으면 됩니다. 이미 준비된 프리미티브를 소비하는 것이 목적이라면, 앞선 섹션들이 더 좋은 출발점입니다.
