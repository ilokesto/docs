# 셀렉터

셀렉터는 `@ilokesto/state`에서 상태를 읽기 위한 주된 메커니즘입니다. 컴포넌트가 스토어의 전체가 아닌 필요한 상태 조각만 구독할 수 있게 하여, 리렌더링 범위를 좁히는 역할을 합니다.

여기서 가장 중요한 개념은 **셀렉터는 읽기(read)를 좁힐 뿐, 쓰기(write)를 좁히는 것이 아니라는 점**입니다. 단일 원시값 필드만 선택하더라도, 반환된 튜플의 두 번째 요소인 "writer"는 여전히 스토어 전체 상태를 다룹니다.

## 기본 사용법

`create`로 생성된 훅은 첫 번째 인자로 셀렉터 함수를 받습니다.

```tsx
const [count, setCount] = useCounter(state => state.count);

// 'count'만 읽었지만, setCount는 여전히 전체 상태를 받고 갱신합니다.
const increment = () => setCount(prev => ({ ...prev, count: prev.count + 1 }));
```

훅은 다음 두 요소를 포함한 튜플을 반환합니다:
1. **선택된 값 (Selected value)**: 셀렉터 함수의 결과값입니다.
2. **Writer**: 스토어 전체를 위한 `setState` 또는 `dispatch` 함수입니다.

## 메모이제이션 및 비교 모델

`@ilokesto/state`는 스토어의 내부 스냅샷 주기와 결합된 "런타임 우선" 메모이제이션 모델을 사용합니다. 별도의 `shallow`나 `isEqual` 같은 공개 비교 API를 제공하지 않는데, 이는 스토어의 구조적 공유(structural sharing)를 기반으로 하기 때문입니다.

- **비교 로직**: 스토어가 업데이트될 때, 훅은 **이전 스토어 스냅샷**과 **다음 스토어 스냅샷**을 깊은 비교(`deepCompare`)를 통해 대조합니다.
- **재사용 동작**: 내부 스냅샷들이 깊게 같다면, 이전 선택값을 그대로 재사용합니다. 이를 통해 셀렉터에서 객체나 배열을 반환하더라도 실제 데이터가 변경되지 않았다면 리렌더링이 발생하지 않도록 보장합니다.

## "셀렉터 없음" 경로 (No-Selector Path)

셀렉터 없이 훅을 호출하면 **전체 스토어 구독** 모드로 동작합니다.

```tsx
// 모든 상태 변화를 구독합니다. 스토어의 어떤 값이든 바뀌면 컴포넌트가 리렌더링됩니다.
const [state, setState] = useStore();
```

작은 스토어나 최상위 프로바이더에서는 편리할 수 있지만, 성능이 중요한 컴포넌트에서는 가급적 피해야 합니다.

## 셀렉터 식별자 및 초기화 (Identity and Resets)

훅은 사용자가 전달한 셀렉터 함수의 식별자(identity)를 추적합니다. 셀렉터 함수 자체가 변경되면(예: 컴포넌트 내부에서 props를 캡처하는 인라인 함수로 정의된 경우), 훅은 내부 구독 상태를 초기화할 수 있습니다.

최상의 성능을 위해 다음을 권장합니다:
- 셀렉터를 안정적으로 유지하십시오 (props에 의존하지 않는다면 컴포넌트 외부에서 정의).
- props에 의존해야 하는 셀렉터라면, 해당 props가 안정적인지 확인하거나 셀렉터가 매번 재평가됨을 인지하고 사용하십시오.

## SSR 및 초기 상태

Server-Side Rendering (SSR) 환경에서 셀렉터는 `getInitialState()`와 상호작용합니다. 훅은 스토어 생성 시 제공된(또는 SSR 하이드레이션을 통해 주입된) 초기 상태를 사용하여 첫 번째 스냅샷을 생성합니다. 덕분에 클라이언트의 첫 번째 렌더링 결과가 서버 출력과 일치함을 보장합니다.

## 파생 상태 (Derived State)

셀렉터 내부에서 직접 파생된 값을 계산할 수 있습니다.

```tsx
const [isOverLimit] = useCounter(state => state.count > 100);
```

훅은 셀렉터의 **결과값**이 변경될 때만 리렌더링을 트리거하므로, `isOverLimit`은 카운트가 임계값을 넘을 때만 리렌더링을 일으키며 단순히 카운트가 증가할 때는 리렌더링을 발생시키지 않습니다.

## 흔히 하는 실수

1. **좁은 쓰기(Narrow Writes) 기대**: `state.count`를 선택했으니 `setCount(5)`가 동작할 것이라고 생각하는 경우입니다. Writer는 항상 `create`에서 정의된 전체 스토어 형식을 따릅니다.
2. **셀렉터 내의 무거운 로직**: selector 경로는 최적화되어 있지만 공짜는 아닙니다. store snapshot이 달라지면 selector는 다시 평가됩니다. selector는 가볍고 예측 가능하게 유지하는 편이 좋습니다.
3. **순환 참조**: 특정 값을 읽으면서 동시에 그 값을 무한히 루프시키는 방식으로 쓰는 셀렉터를 정의하는 경우입니다.
