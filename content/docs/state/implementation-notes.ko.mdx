# 구현 참고 사항 (Implementation Notes)

이 페이지에서는 라이브러리의 내부 아키텍처와 구현 선택 사항을 문서화합니다. 고급 사용자 및 잠재적인 기여자를 위한 내용입니다.

## 훅 구현 (Hook Implementation)

`@ilokesto/state`의 핵심은 `create`가 반환하는 훅이며, 내부적으로 `createUseState`를 통해 `useSyncExternalStore`를 감쌉니다.

- **셀렉터 로직 (Selector Logic)**: 각 훅 호출은 스토어에 대한 구독을 생성합니다.
- **서버 스냅샷 (Server Snapshot)**: `getServerSnapshot`은 `store.getInitialState()`에서 파생되며, 이를 통해 SSR이 원래 생성자 초기값과 맞춰집니다.

## 비교 및 성능 (Comparison and Performance)

- **스토어 계층 (Store Layer)**: 기반이 되는 `Store`는 여전히 `Object.is`를 사용하여 애초에 리스너에게 알릴지 결정합니다.
- **상태 계층 (State Layer)**: `createUseState` 내부에서는 이전 스토어 스냅샷과 다음 스토어 스냅샷을 `deepCompare`로 비교합니다. 깊게 같다고 판단되면 이전 선택 값이 재사용됩니다.
- **별도 equality API 없음**: 현재 공개 API에는 selector equality 함수를 따로 받는 경로가 없습니다.

## Reducer 모드 (Reducer Mode)

`create(reducer, initialState)`를 사용하면, 라이브러리는 내부적으로 `getStore` 내의 미들웨어 인젝션을 사용합니다.

- **액션 디스패치 (Action Dispatching)**: 반환된 `dispatch` 함수는 미들웨어 체인을 통해 액션을 전송하며, 리듀서가 최종적으로 이를 처리하여 새로운 상태를 생성합니다.
- **액션 이름 기록 (Action Name Stamping)**: reducer dispatch 중에는 `getStore`가 `store.actionName`을 잠시 붙여 두고, logger와 devtools가 그 값을 사용합니다.
- **확장성**: 동일한 미들웨어 시스템을 사용하므로, 리듀서 기반 스토어를 로깅, 영속화 및 검증과 결합할 수 있습니다.

## 쓰기 캐시 (영속화) (Write Cache)

과도한 입출력(I/O)을 피하기 위해 `persist` 미들웨어는 쓰기 캐시를 유지합니다.

- **중복 제거 (Deduplication)**: 새로운 상태와 마지막으로 기록된 상태를 비교합니다. 두 상태가 동일하면 쓰기 작업을 건너뜁니다.
- **디바운스 쓰기 (Debounced Writes)**: `persist`를 `debounce`와 결합하여 저장소 부하를 더욱 줄일 수 있습니다.
