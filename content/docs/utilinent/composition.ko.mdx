# 컴포지션

컴포지션 유틸리티는 하나의 구체적인 문제를 해결합니다. 부모 컴포넌트가 동작이나 prop을 제공하고 싶지만, 실제 DOM 요소는 자식이 그대로 가져야 하는 경우입니다.

## `Slot`

`Slot`은 새 래퍼를 하나 더 렌더하는 대신, 자식을 clone해서 prop을 그 자식에 병합합니다.

```tsx
import { Slot } from '@ilokesto/utilinent';

function Button({ asChild, ...props }: { asChild?: boolean } & React.ButtonHTMLAttributes<HTMLButtonElement>) {
  const Component = asChild ? Slot : 'button';
  return <Component className="btn" {...props} />;
}
```

이것이 대표적인 “as child” 패턴입니다.

## `Slottable`

자식 노드가 여러 개 있을 때, 그중 어떤 자식이 병합 대상이 되어야 하는지 표시하고 싶다면 `Slottable`을 사용합니다.

```tsx
import { Slot, Slottable } from '@ilokesto/utilinent';

function ButtonWithIcon({ children }: { children: React.ReactNode }) {
  return (
    <Slot className="btn-with-icon">
      <span aria-hidden>★</span>
      <Slottable>{children}</Slottable>
    </Slot>
  );
}
```

## 실제 병합 규칙

병합 동작은 꽤 명확한 규칙을 따릅니다.

- 이벤트 핸들러는 child 쪽을 먼저 호출하고 slot 쪽을 나중에 호출하며,
- `style` 객체는 병합되고,
- `className`은 이어 붙여지고,
- ref는 compose되어 양쪽이 계속 동작합니다.

유효한 element target이 없으면 `Slot`은 `null`을 반환합니다.

## 언제 이 도구를 쓰면 좋은가

추가 래퍼 요소가 semantics나 스타일링을 망치지만, 부모가 동작은 제공해야 할 때 `Slot`이 가장 잘 맞습니다.

다음 문제가 컴포지션이 아니라 proxy 시스템 자체를 확장하는 것이라면 [확장성](./extensibility)을 이어서 읽으면 됩니다.
