# SSR 및 초기 상태

`@ilokesto/state`의 SSR 지원은 나중에 덧붙인 기능이 아닙니다. 훅이 결국 `useSyncExternalStore`를 사용하기 때문에, React가 요구하는 server snapshot 경로를 자연스럽게 갖게 됩니다.

## 중요한 구분: 현재 상태와 초기 상태

store 계층에서는 이 둘이 다릅니다.

- `getState()`는 현재 살아 있는 값이고,
- `getInitialState()`는 생성자 시점의 값입니다.

`@ilokesto/state`는 server snapshot 경로에서 두 번째 값을 사용합니다.

## 하이드레이션의 기본 모양

```ts
// 서버 사이드
const initialData = { count: 42 };

// 클라이언트 사이드
const useCounter = create(initialData);
```

## `getServerSnapshot`

내부적으로 훅은 React에 `store.getInitialState()`를 `getServerSnapshot`으로 넘깁니다. 그래서 서버에서 렌더링된 트리와 원래 store 초기값이 하이드레이션 시점에 맞춰집니다.

이 때문에 React가 붙기 전에 store를 먼저 바꾸는 코드는 하이드레이션 감각을 예상보다 크게 바꿀 수 있습니다.

## 권장 사항

1. **서버에서는 요청마다 store를 새로 만들기**: 한 사용자의 상태가 다른 요청으로 새지 않게 해야 합니다.
2. **명시적인 초기 데이터에서 하이드레이션하기**: 우연한 클라이언트 쪽 변경에 기대지 않는 편이 좋습니다.
3. **영속화와의 우선순위를 정하기**: persisted client state와 server-provided initial state가 충돌할 수 있으므로, 어느 쪽이 우선인지 분명해야 합니다.
4. **명시적인 bootstrap 지점을 선호하기**: 직렬화된 데이터나 명확한 초기 store factory가 좋습니다.
