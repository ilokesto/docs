# Patterns

`@ilokesto/state` is flexible and allows for several different architectural patterns. Choose the one that best fits your application's needs.

## Separate Actions

Keep your state definition and your update logic separate to improve testability and keep store setup simple.

```ts
// stores/counter.ts
export const useCounter = create({ count: 0 });

// actions/counter.ts
import { useCounter } from '../stores/counter';

export const increment = () => {
  useCounter.writeOnly()(state => ({ count: state.count + 1 }));
};
```

## Derived State (Memoization)

Use simple selectors for derived state. For expensive calculations, memoize them outside of React.

```ts
const useStore = create({ items: [] });

// Simple derived state
const useTotalItems = () => useStore(state => state.items.length);

// Expensive calculation (memoized)
const selectExpensiveData = (state) => {
  // calculate something heavy...
};
const useExpensiveData = () => useStore(selectExpensiveData);
```

## Modular Stores

Instead of one giant store, use several smaller, focused stores. This keeps your application modular and makes it easier to manage state transitions.

```ts
// auth.ts
export const useAuth = create({ user: null });

// products.ts
export const useProducts = create({ list: [] });
```

## Combining with Effects

You can trigger side effects directly from your actions or from within components using `useEffect`.

```ts
// Action with side effect
export const fetchUser = async (id: string) => {
  const user = await api.getUser(id);
  useAuth.writeOnly()({ user });
};
```
