# 패턴

API 자체는 작지만, 실제 프로젝트에서는 이걸 어떻게 조직하느냐가 더 중요합니다. 대부분의 앱은 `create()` 호출 하나보다도 액션을 어디에 둘지, 스토어를 몇 개로 나눌지, 로직을 컴포넌트 안에 둘지 바깥으로 뺄지에서 더 많이 흔들립니다.

## store 정의와 액션은 가깝게 두되, 같은 파일로 뭉개지지 않게 한다

흔한 패턴은 hook 정의는 한 파일에 두고, 액션은 인접한 sibling 모듈에서 export하는 것입니다. 이렇게 하면 `create()` 자체는 작게 유지하면서도, 실제로는 더 읽기 좋은 상태 API를 만들 수 있습니다.

```ts
// stores/counter.ts
export const useCounter = create({ count: 0 });

// actions/counter.ts
import { useCounter } from '../stores/counter';

export const increment = () => {
  useCounter.writeOnly()(state => ({ count: state.count + 1 }));
};
```

이 패턴은 여러 컴포넌트가 같은 쓰기 경로를 공유할 때 특히 유용합니다.

## 파생 상태

파생 상태가 필요할 때 첫 선택지는 대개 selector여야 합니다. store에 파생 필드를 하나 더 넣는 쪽보다 읽는 쪽에서 계산하는 편이 더 단순한 경우가 많습니다.

```ts
const useStore = create({ items: [] });

// 간단한 파생 상태
const useTotalItems = () => useStore(state => state.items.length);

// 비용이 큰 계산은 하나의 selector 뒤에 둔다
const selectExpensiveData = (state) => {
  // 무거운 계산 작업 수행...
};
const useExpensiveData = () => useStore(selectExpensiveData);
```

계산 비용이 정말 크다면 selector를 재사용 가능한 함수로 빼고, 컴포넌트는 그 구현 세부 사항을 모르게 두는 편이 좋습니다.

## 모듈형 스토어

`@ilokesto/state`는 하나의 거대한 전역 상태 트리를 강제하지 않습니다. 실제로는 여러 개의 작은 store가 하나의 과도하게 큰 store보다 더 오래 견디는 경우가 많습니다.

```ts
// auth.ts
export const useAuth = create({ user: null });

// products.ts
export const useProducts = create({ list: [] });
```

도메인이 서로 다른 속도로 바뀌거나, 서로 다른 화면에 속해 있다면 나누는 편이 대개 낫습니다.

## 효과와 결합

`writeOnly()`를 컴포넌트 밖에서 쓸 수 있기 때문에, effectful workflow를 store 정의 안에 섞기보다 액션 주변에 두는 식으로 구성하기 쉽습니다.

```ts
// 사이드 이펙트가 있는 액션
export const fetchUser = async (id: string) => {
  const user = await api.getUser(id);
  useAuth.writeOnly()({ user });
};
```

이 방식은 “로딩 로직까지 store factory 안에 모두 밀어 넣는” 구조보다 경계를 더 깔끔하게 만들어 줍니다.

## 간단한 판단 기준

- 항상 함께 바뀌는 상태라면 하나의 store로 두고,
- 도메인이 독립적이면 store를 나누고,
- 파생 값은 새 필드를 추가하기 전에 selector로 먼저 해결하고,
- 같은 쓰기가 여러 컴포넌트에서 반복되면 액션 뒤로 숨기는 것이 좋습니다.
