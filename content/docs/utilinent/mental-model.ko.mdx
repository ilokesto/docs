# 멘탈 모델

Utilinent는 **렌더링 로직 계층**으로 생각하면 가장 이해하기 쉽습니다. 상태를 소유하지도 않고, 데이터 패칭을 대신하지도 않으며, 스타일 시스템도 아닙니다. 반복해서 나타나는 JSX 로직 패턴에 더 분명한 모양을 주는 패키지입니다.

## 이 패키지는 크게 세 가지 일을 한다

### 1. 자주 쓰는 제어 흐름에 이름을 붙인다

`Show`, `Switch`, `Match`, `For`, `Repeat`, `Mount`가 존재하는 이유는, 조건 분기와 반복이 raw JSX 안에 그대로 남으면 빠르게 퍼지고 흐려지기 때문입니다.

### 2. 가시성을 렌더링 입력으로 바꾼다

`Observer`, `Slacker`, `useIntersectionObserver`는 viewport 상태가 렌더링에 영향을 주도록 만들되, IntersectionObserver 수명주기를 매번 손으로 짜지 않게 해줍니다.

### 3. 래퍼 없이 동작을 자식에 붙인다

`Slot`, `Slottable`은 부모가 동작을 소유하지만 실제 DOM 요소는 자식이 가져야 하는 컴포지션 문제를 해결합니다.

## 선언적이라는 말이 마법을 뜻하는 것은 아니다

이 헬퍼들은 여전히 평범한 React 컴포넌트와 훅입니다.

- `Show`는 여전히 truthiness 규칙을 따르고,
- `Switch`는 첫 번째로 매칭된 분기만 렌더링하며,
- `For`는 결국 배열을 순회하고,
- `Mount`는 children을 해석해서 보이는 결과로 만듭니다.

가치는 새로운 런타임에 있는 것이 아니라, **렌더링 규칙 자체가 JSX에 드러난다**는 데 있습니다.

## 왜 프록시 패턴이 필요한가

여러 export가 proxy-backed인 이유는 `Show.div`, `For.ul` 같은 형태를 가능하게 하기 위해서입니다.

이 프록시 계층은 두 가지 일을 합니다.

- intrinsic tag variant를 만들고,
- category-specific plugin과 base plugin 확장을 받을 수 있게 합니다.

그래서 Utilinent는 단순한 헬퍼 모음이면서도, 동시에 확장 가능한 렌더링 툴킷이 됩니다.

## 어떤 프리미티브를 고르면 좋은가

- 분기 규칙 자체에 이름이 필요하면 **조건부 프리미티브**
- fallback과 반복이 함께 움직이면 **컬렉션 프리미티브**
- viewport나 promise 해석이 출력을 결정하면 **비동기/가시성 프리미티브**
- 래퍼 요소가 API를 망치기 시작하면 **컴포지션 헬퍼**

여기서부터는 대부분 [조건부 렌더링](./conditional)이나 [비동기 및 가시성](./async)으로 이어서 읽으면 자연스럽습니다.
