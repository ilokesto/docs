# 미들웨어

`@ilokesto/state`의 미들웨어는 React 컴포넌트를 감싸지 않습니다. 대신 기반이 되는 store 자체를 감쌉니다. 이 차이가 중요한 이유는, React가 결과를 보기 전에 store 단계에서 업데이트 흐름이 바뀌기 때문입니다.

모든 내장 미들웨어는 `@ilokesto/state/middleware` 엔트리포인트에서 가져올 수 있고, `pipe()`와 함께 자연스럽게 조합되도록 설계되어 있습니다.

```ts
import { create } from '@ilokesto/state';
import { logger, persist } from '@ilokesto/state/middleware';
import { pipe } from '@ilokesto/state/utils';

const useStore = create(
  pipe(
    { count: 0 },
    logger(),
    persist({ local: 'my-app-storage' })
  )
);
```

## 조합 패턴

대부분의 미들웨어는 두 가지 호출 스타일을 지원합니다:
1. **직접 호출**: `middleware(store, options)`
2. **커링된 호출**: `middleware(options)(store)`

store를 선언적으로 조합할 때는 대체로 커링된 형태가 가장 읽기 좋습니다.

## 내장 미들웨어

각 미들웨어는 store 계층에서 서로 다른 역할을 맡습니다.

- [**Logger**](./logger): 콘솔에 상태 변경 내용을 기록합니다.
- [**DevTools**](./devtools): 스토어를 Redux DevTools에 연결합니다.
- [**Persist**](./persist): `localStorage`, `sessionStorage` 또는 쿠키에 상태를 동기화합니다.
- [**Debounce**](./debounce): 빠른 업데이트를 하나의 지연된 flush window로 묶습니다.
- [**Validate**](./validate): Standard Schema v1을 사용하여 상태 업데이트를 검증합니다.

## 실행 순서

`pipe()`를 사용할 때 미들웨어는 위에서 아래로(왼쪽에서 오른쪽으로) 적용됩니다. 이 순서는 단순한 표기 문제가 아니라 실제 update path를 바꿉니다.

예를 들어 `logger()`를 `debounce()` 앞에 두면 모든 업데이트 시도를 기록하고, 뒤에 두면 지연된 flush 결과만 기록합니다.

미들웨어 하나만 쓸 거라면 해당 페이지로 바로 들어가면 되고, 여러 개를 조합할 거라면 기능보다 먼저 순서를 생각하는 편이 좋습니다.
